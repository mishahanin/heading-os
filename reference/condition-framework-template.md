# Condition Framework Template

> A practical tool for Tribe leaders to define and maintain operational states for their areas. Part of the 31C Operational State Model.
> Last Updated: 2026-04-18

---

## Purpose

This template helps you define the operational state for your function — the condition you maintain that makes outcomes inevitable, rather than the targets you chase that create fragility.

An operational state is not a goal in different language. It's a fundamentally different way of operating. A goal says "achieve X by Y date." An operational state says "maintain the condition where X happens as a natural output of how we work."

---

## How to Use This Template

1. Copy the template section below
2. Fill in each field honestly — aspiration is useful, but self-deception is drift
3. Review with your State Check cadence (see State Check Guide)
4. Update as the state evolves — this is a living document, not a contract

Time to complete: 10-15 minutes. If it takes longer, you're overthinking it. If it takes less, you're not being honest enough.


## Filled Example: Product Development (ODUN.ONE)

### Function / Area

Product Development — ODUN.ONE Platform

### Current Reality

Engineering operates in focused sprints with direct deployment feedback from early clients in the home region. The platform is post-commercial launch (February 1, 2026), transitioning from build-to-launch into build-to-scale. Client feedback loops exist but are not yet formalized across all deployment environments. Technical debt from the rapid build phase exists in specific modules. The Research Lab provides AI/ML capabilities but the translation pipeline from research to product varies in speed.

### Operational State

We operate in a condition where every release makes the previous one look primitive and client feedback flows directly into engineering — so that ODUN.ONE's capability compounds with each deployment cycle, and the platform stays years ahead of any legacy DPI vendor or new entrant.

### You Know You're in This State When...

1. Engineering velocity is measured in deployable capability delivered, not story points completed
2. Client feedback from active deployments reaches the engineering team within hours, not weeks
3. The product roadmap is shaped by real sovereign intelligence needs from deployment experience, not by competitor feature lists
4. Technical debt is resolved as it's discovered, not deferred to a future cleanup sprint
5. Every release includes at least one capability that wasn't possible in the previous version

### You've Drifted When...

1. Releases are defined by calendar dates rather than capability readiness
2. Engineering is building features without recent exposure to a live ODUN.ONE deployment
3. Technical debt is accumulating faster than it's being resolved — the codebase is getting heavier, not leaner
4. Features ship to production without having been validated in a deployment environment first
5. The roadmap starts mirroring competitor feature sets instead of leading sovereign intelligence capability

### Course Corrections

- Reconnect engineering to deployment: if engineers haven't seen ODUN.ONE in a client environment recently, that's drift — fix it before fixing anything else
- Shift release criteria from calendar to capability: "ready when the state of readiness is achieved" replaces "ready because the date arrived"
- Audit technical debt honestly: create visibility, set a sustainable resolution rhythm, and treat unresolved debt as drift accumulating silently
- Strengthen the Lab-to-product pipeline: if research breakthroughs aren't reaching ODUN.ONE within a quarter, the translation mechanism needs adjustment

### Core Principle Connection

- **Deliver Under Pressure**: This state ensures the product organization performs its best work when stakes are highest — not by working harder, but by maintaining a cadence where pressure reveals quality
- **Data Sovereignty Always**: Every architectural decision, every feature, every integration is filtered through sovereign control — the state ensures this is never compromised for speed

### State Check Frequency

Weekly — product development moves fast enough that drift can establish itself within a single sprint cycle.

## Common Mistakes

### 1. Disguising Goals as States

**Wrong:** "Operate in a state where we close 10 deals per quarter."
That's a quota wearing a trenchcoat. The number gives it away — if you hit 10, do you stop? If you hit 8, did you fail?

**Right:** "Operate in a rhythm where pipeline replenishes faster than it depletes and every deal is closed on the strength of demonstrated value."
This is a condition. It doesn't end at a number. It describes how you operate, not what you extract.

### 2. Writing Aspirational Fiction Instead of Operational Reality

The "Current Reality" section must be honest. If you write "everything is going well, we just need to keep going," you haven't diagnosed — you've performed. A doctor who tells every patient they're healthy isn't a good doctor. Read the actual symptoms.

### 3. Making Indicators Unmeasurable

**Wrong:** "You know you're in this state when the team feels good about the product."
Feelings are not observable indicators. What does "feels good" look like in behavior?

**Right:** "You know you're in this state when engineers voluntarily demo new capabilities to the Tribe before being asked."
That's observable. That's behavioral. That's a real signal.

### 4. Course Corrections That Are Actually New Goals

**Wrong:** "Course correction: achieve 95% test coverage."
That's a goal. It has a number and an endpoint.

**Right:** "Course correction: integrate automated testing into the development rhythm so that quality assurance is continuous, not periodic."
That's an adjustment to how you operate.

### 5. Confusing State Check Frequency with Reporting Frequency

State Checks are diagnostic, not administrative. If you're checking your state weekly but nothing changes between checks, either your state is perfectly maintained (rare) or you're checking without actually looking. Match frequency to the rate at which your function's sea state changes.
